## 1. Project Overview

**App name:** Mise en place  
**Type:** Restaurant staff scheduling web app  
**Stack:** React (frontend) + Supabase (backend/database) + TanStack Query (data fetching)
**Scheduling engine:** Python microservice using Google OR-Tools (CP-SAT solver) + Claude AI for conflict resolution, exposed via FastAPI

The app replaces a manual paper schedule (a monthly grid: rows = employees, columns = days) that the restaurant manager currently fills in by hand.

## 2. Scheduling Architecture — Hybrid OR-Tools + Claude AI

**This is the most important architectural decision in the project. Do not simplify it.**

The scheduling pipeline has two stages:

```
React collects data from Supabase
        ↓
POST /generate-schedule (FastAPI)
        ↓
Stage 1 — OR-Tools tries to solve normally
        ↓
  Found? ──YES──→ Return assignments (status: "ok")
        ↓ NO
Stage 2 — diagnose_conflict() identifies WHY it failed
        ↓
Claude AI receives the conflict report
        ↓
Claude suggests which constraint to relax + why (in French)
        ↓
OR-Tools retries with the relaxed constraint
        ↓
  Found? ──YES──→ Return assignments + Claude's message (status: "ok_with_compromise")
        ↓ NO
Return error + Claude's message (status: "error")
```

### Why this split?
- **OR-Tools** handles mathematical correctness: it guarantees all hard constraints are satisfied or tells you it's impossible. It cannot make judgment calls.
- **Claude AI** handles conflict resolution: when constraints clash, it reasons about the situation in human terms, suggests the least disruptive compromise, and explains it in plain French to the manager.
- Claude AI is **never** used to generate the raw schedule — only to resolve conflicts. This ensures the final schedule is always mathematically valid.

### ScheduleResult shape (returned by generate_schedule())
```python
@dataclass
class ScheduleResult:
    status: str           # "ok" | "ok_with_compromise" | "error"
    assignments: dict     # { (employee_id, day_index, shift_id): True }
    manager_message: str  # Claude's French explanation (only on compromise or error)
    conflict: ConflictReport  # Raw conflict details (only on failure)
```

### AI Relaxation types
When Claude suggests a fix, it returns one of these machine-readable instructions:
```json
{ "type": "reduce_headcount",       "shift_id": 0, "day_of_week": 5, "role": "cook", "new_value": 1 }
{ "type": "ignore_unavailability",  "employee_id": 3, "date_index": 6 }
{ "type": "extend_hours",           "employee_id": 2, "extra_hours": 4 }
{ "type": "no_fix_possible",        "reason": "..." }
```

## 3. Core Domain Concepts

| Concept | Description |
|---|---|
| **Schedule Month** | One per calendar month. Status: `draft` or `published`. |
| **Employee** | Name, role, weekly contract hours, team (A or B). |
| **Role** | One of: `cook`, `waiter`, `barman` |
| **Team** | `A` or `B` — used for morning/afternoon rotation (soft constraint, rule TBD by client) |
| **Shift Type** | Named shift with start/end time, closing flag, default headcount per role. |
| **Shift Requirement** | Overrides headcount for a specific (shift × day-of-week). e.g. Saturday needs 2 cooks. |
| **Unavailability** | Employee-submitted date they cannot work. |
| **Assignment** | One employee × one shift type × one date. Generated by algorithm. |

## 5. Scheduling Constraints

### Hard Constraints (must never be violated)
| ID | Rule |
|---|---|
| C1 | No assignment on unavailable dates |
| C2 | Each shift meets headcount per role — varies by day-of-week via `shift_requirements` |
| C3 | Each employee has exactly 2 consecutive days off per 7-day week |
| C4 | Weekly hours must not exceed employee's `weekly_contract_hours` |

### Soft Constraints (optimize toward)
| ID | Rule |
|---|---|
| C6 | Closing shifts balanced across employees (minimize max−min) |
| C7 | Team A/B alternates morning/afternoon weekly — **rule not yet confirmed by client, implement as soft preference** |

### Conflict Resolution Flow
1. OR-Tools fails → `diagnose_conflict()` runs lightweight checks to identify which constraints are causing infeasibility
2. `ConflictReport` is sent to Claude with full employee/shift context
3. Claude returns `manager_message` (French, 2-3 sentences) + machine-readable `relaxation`
4. OR-Tools retries with the relaxation applied
5. If still fails → surface error + Claude's message to manager for manual resolution

## 7. React + TanStack Query Integration

Supabase client is initialized at: **`src/lib/supabase.ts`** ← verify this path

```typescript
// Collect data from Supabase, call Python service, write results back
const { mutate: generateSchedule, isPending, isError } = useMutation({
  mutationFn: async (scheduleMonthId: string) => {
    // 1. Fetch all data from Supabase
    const [employees, shiftTypes, shiftRequirements, unavailabilities, month] =
      await Promise.all([
        supabase.from('employees').select('*'),
        supabase.from('shift_types').select('*'),
        supabase.from('shift_requirements').select('*'),
        supabase.from('unavailabilities').select('*').eq('schedule_month_id', scheduleMonthId),
        supabase.from('schedule_months').select('*').eq('id', scheduleMonthId).single(),
      ])

    // 2. Call Python microservice
    const response = await fetch(`${process.env.VITE_SCHEDULER_URL}/generate-schedule`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ ...month.data, employees: employees.data, ... })
    })
    const result = await response.json()

    // 3. On success, write assignments back to Supabase
    if (result.status !== 'error') {
      await supabase.from('assignments').insert(result.assignments)
    }

    return result
  },
  onSuccess: (result) => {
    if (result.status === 'ok_with_compromise') {
      // Show Claude's French message in a toast/banner
      showCompromiseMessage(result.manager_message)
    }
    queryClient.invalidateQueries({ queryKey: ['assignments', scheduleMonthId] })
  }
})
```

## 8. UI Requirements

- Primary view: **grid** — rows = employees, columns = days of month
- Each cell: shift label + compact times (e.g. `09h–16h`)
- **TOTAL column** on the right: hours worked + closing shift count per employee
- Cell states: assigned / day off / unavailable / manual override (visually distinct)
- **Compromise banner**: when `status === "ok_with_compromise"`, show Claude's `manager_message` prominently above the grid
- Manager can override any cell after generation (`is_manual_override = true`)
