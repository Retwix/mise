## 1. Project Overview

**App name:** Mise en place  
**Type:** Restaurant staff scheduling web app  
**Stack:** React (frontend) + Supabase (backend/database)  
**Scheduling engine:** Python microservice using Google OR-Tools (CP-SAT solver), exposed via FastAPI  

The app replaces a manual paper schedule (a monthly grid: rows = employees, columns = days) that the restaurant manager currently fills in by hand.

---

## 2. Core Domain Concepts

| Concept | Description |
|---|---|
| **Schedule Month** | One per calendar month. Has a status: `draft` or `published`. |
| **Employee** | Has a name, a role, a weekly contract hours value, and belongs to a team. |
| **Role** | One of: `cook`, `waiter`, `barman` |
| **Shift Type** | A named shift with a start/end time, a closing flag, and default headcount requirements per role. |
| **Shift Requirement** | Overrides headcount for a specific (shift × day-of-week) combination (e.g. Saturday needs 2 cooks instead of 1). |
| **Unavailability** | An employee-submitted date they cannot work. |
| **Assignment** | Links one employee to one shift type on one specific date. Generated by the algorithm. |

## 4. Scheduling Constraints

These are implemented in `scheduler.py` (Python / OR-Tools). The API receives all data from Supabase and returns a list of assignments.

### Hard Constraints (must never be violated)
| ID | Rule |
|---|---|
| C1 | An employee cannot be assigned on a date they marked as unavailable |
| C2 | Each shift on each day must meet its required headcount per role (cook/waiter/barman) — requirement varies by day of week via `shift_requirements` table |
| C3 | Each employee must have exactly 2 **consecutive** days off per 7-day week |
| C4 | An employee's total hours in a week must not exceed their `weekly_contract_hours` |
| C5 | A day's headcount requirement is day-of-week aware: Saturday/Sunday need more staff, Monday less — configured in `shift_requirements` |

### Soft Constraints (optimize toward, but don't block on)
| ID | Rule |
|---|---|
| C6 | Closing shifts should be balanced as evenly as possible across all employees over the month (minimize max−min closing count) |
| C7 | Employees should alternate between morning-heavy and afternoon-heavy weeks (Team A / Team B rotation) — **rule not fully defined yet by client, implement as a soft preference** |

### Conflict Resolution
If constraints cannot all be satisfied simultaneously:
- Return a **clear error** (do not silently generate a broken schedule)
- Identify **which constraint** was impossible to satisfy if possible
- Let the manager fix it manually in the UI

## 5. Python Scheduling Microservice

**File:** `scheduler.py`  
**Framework:** FastAPI  
**Library:** `ortools` (Google CP-SAT solver)

### Expected API endpoint
```
POST /generate-schedule
```

### Request body (sent by React app)
```json
{
  "schedule_month_id": "uuid",
  "month": 12,
  "year": 2024,
  "start_weekday": 6,
  "num_days": 31,
  "employees": [
    { "id": "uuid", "name": "Alice", "role": "cook", "weekly_contract_hours": 35, "team": "A" }
  ],
  "shift_types": [
    { "id": "uuid", "label": "Morning", "start_hour": 9, "end_hour": 16, "is_closing": false,
      "default_required_cooks": 1, "default_required_waiters": 1, "default_required_barmen": 1 }
  ],
  "shift_requirements": [
    { "shift_type_id": "uuid", "day_of_week": 5, "required_cooks": 2, "required_waiters": 2, "required_barmen": 2 }
  ],
  "unavailabilities": [
    { "employee_id": "uuid", "date": "2024-12-05" }
  ]
}
```

### Response body (on success)
```json
{
  "status": "ok",
  "assignments": [
    { "employee_id": "uuid", "shift_type_id": "uuid", "date": "2024-12-01" }
  ]
}
```

### Response body (on failure)
```json
{
  "status": "error",
  "message": "No valid schedule found. Possible cause: insufficient staff on Saturdays."
}
```

## 6. React App Integration

When the manager clicks **"Generate Schedule"**:
1. React fetches all relevant data from Supabase (employees, shift types, requirements, unavailabilities for that month)
2. Sends a `POST /generate-schedule` request to the Python microservice
3. On success: writes the returned assignments back to the `assignments` table in Supabase
4. On error: displays the error message and lets the manager adjust manually

### UI Requirements (from paper schedule analysis)
- Primary view is a **grid**: rows = employees, columns = days of month
- Each cell shows the shift label + times compactly (e.g. `09:00–16:00`)
- A **TOTAL column** on the right showing hours worked and closing shifts count per employee
- Manager can **manually override** any cell after generation (sets `is_manual_override = true`)
- Cells should be visually distinct for: assigned, day off, unavailable, manual override
- Support a **highlight/flag** on rows with anomalies (e.g. constraint violations after manual edit)
